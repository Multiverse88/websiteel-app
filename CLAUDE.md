# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Is

**EasyLegal** — Indonesian business-legality services platform. Next.js 16 App Router project that markets and sells 12+ legal services (PT/CV establishment, HAKI trademarks, NIB/OSS, ISO certification, PKP tax, visa/KITAS, etc.), runs a content/blog system, a drag-and-drop landing-page builder, and a JWT + TOTP-2FA admin dashboard. Public content is mostly Indonesian; admin UI is Indonesian with a `#990202` crimson theme.

The production domain is `easylegal.my.id` (nginx config, `src/lib/config.ts` `baseUrl`, `.env.example` all point here). `easylegal.id` still appears in a few places (e.g. `company.email` in `config.ts`, some docs) — treat `easylegal.my.id` as authoritative unless told otherwise.

The repository root also contains design/source materials extracted from Figma, one-off `.py`/`.js` migration scripts used to bulk-edit page content, planning/report `.md` files, and a `.qoder/repowiki/` autogenerated reference. Ignore these unless the user is specifically working on design parity, content migration, or wiki content.

## Environment & Database

- `cp .env.example .env` for local dev. Required vars: `DATABASE_URL`, `DIRECT_URL`, `JWT_SECRET`. Optional: `SMTP_*`, `NEXT_PUBLIC_APP_URL`, `MINIO_*` (object storage CDN, see below).
- Database is PostgreSQL (use `host.docker.internal` from the app container to reach a host-installed Postgres, per `docker-compose.yml`).
- Prisma schema: `prisma/schema.prisma`. Migrations live in `prisma/migrations/`. Models: `User`, `Article`, `NewsletterSubscriber`, `NewsletterBroadcast`, `EmailLog`, `SystemSetting`, `ContactSubmission`, `Redirect`, `LandingPage`, `LandingPageLead`. Author relations (`Article.author`, `LandingPage.author`) — `Article.authorId` is optional with `onDelete: SetNull`.
- Seeding (`prisma/seed.ts`) seeds ~200 articles (10 inline + ~190 from `prisma/articles-seed.json`). Skips if articles already exist; use `FORCE_SEED=true` or `--force` to clear and re-seed. Does **not** create admin users or system settings — use `scripts/create-admin.ts` for that.
- The Prisma client is the standard global-singleton pattern in `src/lib/db.ts` (dev hot-reload safe). Some newer Server Actions/pages instantiate `new PrismaClient()` directly instead of importing the singleton — prefer `@/lib/db` for new code.

## High-Level Architecture

**Hybrid rendering, Server-Action backend.** There are no `app/api/*` route handlers (aside from the auto-generated Sentry example) — every mutation goes through Server Actions (`"use server"` files like `src/app/login/actions.ts`, `src/app/dashboard/newsletter/actions.ts`, `src/app/dashboard/artikel/tambah/actions.ts`, `src/app/dashboard/landing-pages/actions.ts`).

```
websiteel-app/src
├── app/                # App Router pages + Server Actions
│   ├── (public)        # /, /tentang-kami, /kontak, /artikel, /artikel/[slug],
│   │                   # /cek-nama, /cek-kbli, /testimoni, /referral-reseller, /kerjasama,
│   │                   # /layanan/*, /home-gads, /newsletter/unsubscribe, /lp/[slug]
│   ├── login/          # /login + 2FA setup/verify Server Actions
│   └── dashboard/      # Protected (see middleware.ts); article CMS, newsletter,
│                       # landing-pages builder, redirect links, profile
├── components/         # Shared (Navbar, Footer, CTA, FAQ, Pricing, …)
│   ├── home/           # Homepage section components (Hero, LayananKami, etc.)
│   ├── layanan/        # Shared service-page templates (e.g. BadanUsahaTemplate)
│   ├── landing-page/   # SectionRenderer + sections/ (Hero, Features, LeadForm, …)
│   │                   # used by both the /lp/[slug] renderer and the builder
│   └── ui/             # shadcn primitives (accordion, button, …)
├── data/               # Static content data (e.g. layanan-badan-usaha.tsx)
├── lib/                # auth.ts, 2fa.ts, db.ts, config.ts, mail.ts, s3.ts,
│                       # rate-limit.ts, metrics.ts, compress-image.ts, utils.ts
└── middleware.ts        # Dashboard JWT auth guard + DB-driven slug redirects
```

### Authentication flow
1. `POST /login` → `loginAction` (Server Action) bcrypt-verifies against `User`, sets a 5-min `pending_2fa_user` cookie, redirects to `/login/setup-2fa` (first time) or `/login/verify-2fa` (returning).
2. 2FA actions live in `src/app/login/actions-2fa.ts` and use helpers in `src/lib/2fa.ts` (`otplib` + `qrcode`).
3. On success, `createSession` (`src/lib/auth.ts`) mints a 24h HS256 JWT (`jose`) into the `admin_token` httpOnly cookie. `src/lib/config.ts`'s `getJwtSecret()` encodes `JWT_SECRET` for both Node and Edge runtimes.
4. `src/middleware.ts` guards `/dashboard`, `/dashboard/:path*` via `jwtVerify`, redirecting to `/login` on failure.
5. Each dashboard Server Component also calls `getSession()` defensively and re-checks `twoFactorEnabled` (see `src/app/dashboard/page.tsx`).
6. `src/lib/rate-limit.ts` provides in-memory login/2FA attempt tracking and lockout (not distributed — fine for single-instance deploy, would need Redis to scale out).

### Middleware also does DB-driven redirects
Beyond the dashboard auth check, `middleware.ts` runs on every `GET`/`HEAD` request: it looks up `pathname` against `Redirect.slug` and 307-redirects to `destination` if found, incrementing `clicks` fire-and-forget. It dynamically `import()`s `@/lib/db` (Prisma is Node-only, can't be a top-level import in the Edge runtime) and swallows DB/runtime errors so the site keeps working if the lookup fails. Admin CRUD for these lives at `/dashboard/links`.

### Service-page pattern
`/layanan/<slug>` pages follow one of two patterns:
- **Template-driven**: pages that share a body template (e.g. `pendirian-badan-usaha/page.tsx` → `BadanUsahaTemplate` reading from `src/data/layanan-badan-usaha.tsx`, with sub-routes under `[jenis]/page.tsx`).
- **Self-contained**: long single-file pages (e.g. `merek-haki/page.tsx`, `nib-oss/page.tsx`, `press-release/page.tsx`) that import the shared `Navbar`/`Footer` via the root layout and compose their own sections.

There is a permanent redirect from `/layanan/pendirian-yayasan` → `/layanan/pendirian-badan-usaha/yayasan` (declared in `next.config.ts`), plus 50+ `rewrites()` aliases mapping ad-campaign landing URLs (`*-gads`, `*-dads`, `*-pmax`, `*-ytads` suffixes) onto real service pages — check there before assuming a URL is dead.

### Landing Page Builder
A separate system from the static `/layanan` pages, for marketing to build/publish standalone pages without a deploy:
- **Model**: `LandingPage` (JSON `sections` blob + `redirectSettings` JSON, `status: "draft" | "published"`) and `LandingPageLead` (captured form submissions).
- **Public render**: `src/app/lp/[slug]/page.tsx` — fetches the page by slug (published only), parses `sections` JSON, renders via `SectionRenderer` (`src/components/landing-page/`), injects the Facebook Pixel script if `pixelId` is set.
- **Builder UI**: `src/app/dashboard/landing-pages/[id]/BuilderClient.tsx` — uses `@dnd-kit` for drag-and-drop section reordering; persists via `updateLandingPageSections` Server Action.
- **Section types**: `src/components/landing-page/sections/` (`HeroSection`, `FeaturesSection`, `BannerSection`, `LeadFormSection`, `TestimonialsSection`, `RichTextSection`); typed in `src/types/landing-page.ts`.

### Article system
- **Listing** (`src/app/artikel/page.tsx`): ISR with `revalidate = 60`.
- **Detail** (`src/app/artikel/[slug]/page.tsx`): ISR (`revalidate = 300`), custom markdown renderer in-file (supports `### headings`, `**bold**`, bullet/numbered lists, `---`); auto-generated ToC (`table-of-contents.tsx`); client-side `view-tracker.tsx` increments `viewCount`.
- **CMS** (`src/app/dashboard/artikel/tambah/` and `[id]/edit/`): client-component with live preview, image upload (to MinIO if configured, else local) or URL mode.

### Image storage (MinIO / S3-compatible CDN)
- `src/lib/s3.ts` wraps `@aws-sdk/client-s3` against a MinIO instance; `isMinioConfigured()` gates on `MINIO_ENDPOINT`/`MINIO_ACCESS_KEY`/`MINIO_SECRET_KEY` being present. When unset, upload flows fall back to local/URL handling.
- Uploaded images go through `uploadToMinio(buffer, filename, contentType, folder)` → public URL under `MINIO_PUBLIC_URL` (served in prod via `cdn.easylegal.my.id`).
- `scripts/sync-images.js` bulk-syncs `public/` static images into the same bucket on container start (HEAD-checks before upload, skips existing); it auto-detects VPS deploy (`IS_VPS`/`.dockerenv`) and switches to the internal MinIO endpoint to avoid routing through Cloudflare.
- `src/lib/compress-image.ts` handles client-side image compression (`browser-image-compression`) before upload.

### Newsletter system
- Subscribe via the widget on article pages → `NewsletterSubscriber` row. Unsubscribe at `/newsletter/unsubscribe`.
- Admin broadcasts from `/dashboard/newsletter` → loops active subscribers, sends via Nodemailer, logs each attempt to `EmailLog` linked to a `NewsletterBroadcast` parent.
- If `SMTP_*` env vars are absent, `src/lib/mail.ts` falls back to a console simulation (`status: "simulated"`).
- `SystemSetting` key/value store drives auto-broadcast on new article creation (see `src/app/dashboard/artikel/tambah/actions.ts`).

### Contact form
- Submissions via `/kontak` → `ContactSubmission` row (`status: "new" | "read" | "replied"`). Admin email notification via Nodemailer.
- Public lead-gen pages (`/kontak`, `/cek-nama`, `/cek-kbli`, `/referral-reseller`, `/kerjasama`) primarily funnel to WhatsApp via `getWhatsAppLink()` (`src/lib/config.ts`), which routes through a `mauorder.online` link rotator rather than a raw `wa.me` link — don't hardcode `wa.me` links in new code, use the helper.

### Redirect links
- Database-driven redirects via the `Redirect` model, resolved in `middleware.ts` (see above). Admin CRUD from `/dashboard/links`.

### Dashboard
- Protected by JWT middleware (`src/middleware.ts`) + defensive `getSession()` in each page.
- Sections: articles (`/dashboard`), newsletter (`/dashboard/newsletter`), landing pages (`/dashboard/landing-pages`), redirect links (`/dashboard/links`), profile (`/dashboard/profile`).
- Each dashboard page starts with `export const dynamic = "force-dynamic";` and an auth check.

### Observability
- `@sentry/nextjs` is wired through `next.config.ts` (`withSentryConfig`), `instrumentation.ts` (registers `sentry.server.config.ts`/`sentry.edge.config.ts` in production only, via `NEXT_RUNTIME`), and `sentry.client.config.ts`.
- `src/lib/metrics.ts` exposes `trackMetric`/`trackDistribution` thin wrappers over `Sentry.metrics`, no-ops outside production.
- A strict `Content-Security-Policy` is generated in `next.config.ts` (stricter in prod — no `unsafe-eval`) allowlisting GTM, YouTube, a self-hosted Typebot instance, Cloudflare, Facebook Pixel/Conversions, and Google Ads domains. Extend this list rather than loosening `script-src` broadly when adding a new third-party embed.

### Animations
- GSAP is the primary animation library; `ScrollManager.tsx` (mounted in root layout) handles smooth-scroll for same-page links + hash navigation on route change using `ScrollToPlugin`.
- `template.tsx` is currently a thin wrapper. Page transitions are largely handled by per-component `useGSAP` hooks (e.g. on the homepage).
- Framer Motion is used for some `whileInView` reveals. Do not introduce a third animation library.

## Docker / Deployment

Multi-stage `Dockerfile` (deps → builder → runner, all `node:20-alpine`, `apk add openssl` for Prisma). Final image runs `npx prisma@6.19.3 migrate deploy && node scripts/sync-images.js && node server.js`, so migrations apply and static images sync to MinIO on every container start. `next.config.ts` sets `output: "standalone"` for small image size.

`docker-compose.yml` runs two services:
- `app` — Next.js container, reads `.env.production`, mounts the `uploads` Docker volume at `/app/public/uploads`.
- `nginx` — `nginx:alpine` reverse proxy, exposes 80/443, mounts host Let's Encrypt certs, shares the `uploads` volume read-only.

PostgreSQL and MinIO are expected to run directly on the host VPS (not in Compose). Inside the app container, `host.docker.internal:host-gateway` resolves to the host — already configured via `extra_hosts` in Compose. `cdn.easylegal.my.id` must CNAME to the VPS IP for public image URLs to resolve.

Key scripts in `websiteel-app/scripts/`:
- `deploy-setup.sh` — one-time VPS bootstrap (Docker, certbot, UFW, backup dir).
- `deploy.sh` — backup → `git pull` → `docker compose build --no-cache` → `up -d` → cleanup → health check.
- `backup-db.sh` — `pg_dump` to `/opt/backups/easylegal_*.sql.gz`, auto-cleans >7 days.
- `sync-images.js` — syncs `public/` static images to MinIO (see above).
- `sync-and-clean.sh`, `optimize-images.sh`, `optimize-db.sh`, `check-ssl.sh`, `audit-security.sh`, `monitor-logs.sh`, `secure-vps.sh` — operational/maintenance helpers.
- `create-admin.ts` — creates the first admin `User` (seeding does not do this).
- `get-2fa-qr.mjs` — prints a 2FA QR code for a user, for local/manual setup.

Nginx config in `websiteel-app/nginx/nginx.conf` is wired to `easylegal.my.id`; it also reverse-proxies `easyai.id` to an unrelated `recruit-app` service — a 502 on that domain is expected if that service isn't running and does not indicate an EasyLegal problem.

## Design / Styling

- Tailwind v4 via `@tailwindcss/postcss` (no `tailwind.config.js`; tokens declared inside `src/app/globals.css` `@theme { … }`).
- Custom color tokens: `primary #D62828`, `primary-light #FEF2F2`, `primary-hover #B91C1C`, admin `crimson #990202`, `whatsapp #25D366`. Custom breakpoint `xs: 375px`.
- Fonts loaded via `next/font/google` in `src/app/layout.tsx`. Only DM Sans (`--font-dm-sans`) is actually imported/used across the app — Plus Jakarta Sans and Inter are referenced in some older docs but not currently wired in.
- Reusable CSS classes defined in `globals.css`: `.glassmorphism`, `.custom-shadow`, `.custom-shadow-lg`, plus `animate-*` keyframes (`slideDown`, `fadeInUp`, `float`, `pulse-subtle`, `fadeIn`, `marquee-left/right`).
- shadcn/ui (style `new-york`, base `neutral`) is installed. Add components with `npx shadcn@latest add <name>`. Custom registries include `@react-bits` (see `components.json`/`.mcp.json`).

## Conventions to Follow

- Path alias `@/*` → `src/*`.
- All new mutation endpoints → Server Actions with `"use server"`, not API routes.
- Dashboard pages start with `export const dynamic = "force-dynamic";` and a defensive `getSession()` + `redirect("/login")` check.
- For long static service pages, prefer a single self-contained `page.tsx` over fragmenting into many components; existing service pages (700–1500 lines each) are the reference style.
- Use `getWhatsAppLink()` from `src/lib/config.ts` for any WhatsApp CTA rather than hardcoding a `wa.me` URL.
- Generated image folders under `public/uploads/` are gitignored (only `.gitkeep` placeholders are kept). Do not commit uploaded assets.
- `.env.production`, `nginx/certs/`, and `*.pem` must never be committed.
- `eslint.config.mjs` ignores `scripts/**` (plain Node scripts using `require()`, not part of the Next.js app) — don't try to make them pass Next's lint rules.
